# 技术选型预警规则

## 🔴 最高优先级规则

### 核心原则

**当一个问题需要第二次修复时，必须停下来分析根因，而不是继续打补丁。**

---

## 📍 触发条件

以下任一情况出现时，**立即停止当前修复方向**，转为根因分析：

1. **同一类错误出现第二次** — 说明第一次修复不彻底
2. **需要加"防御层"来防止某个问题** — 说明根源没解决
3. **一个 hack 引发了新问题，需要另一个 hack 来修** — 说明方向错了
4. **需要用 `window` 全局变量、monkey-patch 等非常规手段** — 说明架构有问题
5. **同一工具在不同场景下都要特殊处理** — 说明该工具与技术栈不兼容

---

## 🚫 禁止的行为

- ❌ 在同一个问题上连续加超过 **1 层** "防御/保障"
- ❌ 用 hack 修 hack（如用 window 全局单例绕过构建问题，再用脚本验证单例是否生效）
- ❌ 遇到不兼容时不告知用户，默默加 workaround
- ❌ 降级/升级依赖时不检查与现有技术栈的兼容性
- ❌ 引入新依赖时不检查其维护状态和与当前栈的兼容性

---

## ✅ 必须执行的动作

当触发条件出现时，**必须停下来执行以下步骤**：

### 第一步：向用户报告技术风险

```
⚠️ 技术选型预警：
- 问题：[具体描述问题]
- 已尝试的修复：[列出已尝试的 hack/workaround]
- 根因分析：[为什么问题反复出现]
- 当前状态：[是否工作，但存在风险]

🔍 根本原因推断：
[分析为什么这个问题会产生]

例如：Vite 6 对 React 18（纯 CJS）的 CJS→ESM 转换有 bug，
会产生两份 React 实例，导致 Context 双实例。
已尝试的 hack：resolve.dedupe + manualChunks + window 全局单例
但这只是 workaround，根本方案应该是：
  1. 降级 Vite 到 5.x（对 CJS 处理成熟）
  2. 或切换到其他构建工具
```

### 第二步：列出根本解决方案（2-3 个）

```
方案 A：降级工具版本 [优点/缺点]
方案 B：替换工具 [优点/缺点]
方案 C：改变架构 [优点/缺点]
```

### 第三步：让用户决定

**不要自己决定继续打补丁**。把选择权交给用户。

### 第四步：执行用户选择的根本方案

## 🔧 技术选型检查清单

在以下情况时**必须执行**检查清单：

1. **引入新依赖前**
2. **升级工具版本前**
3. **选择技术方案时**

### 检查项

- [ ] 该工具/库是否与当前技术栈的**模块格式兼容**（CJS vs ESM）？
- [ ] 该工具是否有与现有技术的**已知兼容性问题**（搜索 GitHub issues）？
- [ ] 主要依赖的 **peer dependency 版本**是否与现有栈匹配？
- [ ] 该工具是否在**积极维护**？最近发布时间是何时？
- [ ] 官方是否已标记为 **deprecated** 或推荐迁移？
- [ ] 如有不兼容风险，**是否已告知用户**？

### 已废弃技术黑名单

某些技术已被官方停止维护，**禁止在新项目中选用**（除非用户知情同意）：

| 技术 | 替代方案 | 状态 |
|------|---------|------|
| Create React App (CRA) | Vite / Next.js | ❌ 停止开发 |
| Moment.js | Day.js / date-fns | ❌ 不推荐 |
| Request (HTTP) | Axios / node-fetch / undici | ❌ 停止维护 |
| TSLint | ESLint | ❌ 已迁移 |
| Bower | npm / yarn / pnpm | ❌ 已弃用 |

## 📚 历史教训

### 本项目的案例（Vite + React CJS 问题）

**症状**：React Context "must be used within Provider" 报错

**错误的修复路径**：
1. 第 1 层 hack：加 `resolve.dedupe`（辅助作用，但不够）
2. 第 2 层 hack：加 `resolve.alias` 强制指向 React（没用）
3. 第 3 层 hack：加 `manualChunks`（进一步恶化问题）
4. 第 4 层 hack：加 `window` 全局单例（这时才意识到根源是 Vite 6 的 bug）
5. 第 5 层 hack：创建验证脚本 `verify-bundle.cjs`（复杂度爆炸）

**浪费时间**：数小时的反复尝试

**正确的修复**：
- 停下来分析：Vite 6 对 CJS React 的 CJS→ESM 转换有 bug
- 根本方案：降级到 Vite 5.x（对 CJS 处理成熟）
- 结果：所有 hack 删除，配置干净，问题彻底解决

**关键洞察**：
> 如果第一次报错时就停下来分析根因，5 分钟就能解决。反复打补丁浪费了数小时。

## 💡 判断流程

遇到问题时，按这个流程判断是否需要停下来分析根因：

```
1. 这是第几次修这个问题？
   ├─ 第 1 次 → 尝试修复，记录方案
   └─ 第 2 次+ → 🛑 停下，分析根因
   
2. 修复时是否在加"保护层"？
   ├─ 是 → 🛑 停下，这说明根源没解决
   └─ 否 → 继续修复
   
3. 修复方案是否超过"正常范围"？
   ├─ 是（如需要全局变量、脚本验证等）→ 🛑 停下，方向可能错了
   └─ 否 → 继续修复
   
4. 如果停下了：
   → 分析根因
   → 向用户报告 + 列出方案
   → 等用户决定
```

## ⚠️ 常见误区

### 误区 1：觉得"反正能工作"就不用分析根因

**错误**：Hack 方案虽然能工作，但埋下了隐患：
- 代码复杂度增加
- 下次修改时更容易引发新问题
- 其他开发者难以理解
- 维护成本大幅提高

**正确做法**：停下来分析根因，做根本解决。

### 误区 2：不敢告诉用户有技术风险

**错误**：默默打补丁，用户不知道存在问题，将来可能爆发。

**正确做法**：明确告诉用户："这里有技术风险，可能的方案是..."，让用户知情决策。

### 误区 3：用户没时间等，就继续打补丁

**错误**：短期快速，但后期债务很大。

**正确做法**：告诉用户"快速修复但有风险"与"彻底修复但需要时间"的权衡，让用户选。

## ✅ 检查清单

在完成修复前，必须确认：

- [ ] 这是同一问题的第几次修复？
- [ ] 是否在加"防御层"或 hack？
- [ ] 根因是什么？
- [ ] 已列出 2-3 个根本方案吗？
- [ ] 用户已知晓技术风险吗？
- [ ] 用户确认了修复方案吗？
